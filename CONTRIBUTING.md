## üìã Issue Guidelines

### Before Creating an Issue

1. **Search existing issues** - Check if your issue already exists
2. **Use the latest version** - Ensure you're running the most recent release
3. **Check documentation** - Review existing docs and guides
4. **Gather information** - Collect relevant details before reporting

### Writing Good Issues

#### For Bug Reports
- **Clear, descriptive title** - Summarize the problem in a few words
- **Environment information** - OS, browser, EVE client version, tool version
- **Reproduction steps** - Step-by-step instructions to reproduce the issue
- **Expected vs actual behavior** - What should happen vs what actually happens
- **Screenshots or logs** - Visual evidence or error messages
- **Workarounds** - Any temporary solutions you've found

#### For Feature Requests
- **Problem statement** - What problem does this solve?
- **Proposed solution** - Your idea for solving it
- **Use cases** - Real-world scenarios where this would help
- **Alternatives** - Other solutions you've considered
- **Priority** - How important is this to you and why?

#### For Module Requests
- **Module purpose** - What would this tool accomplish?
- **Target users** - Who would benefit from this module?
- **Core features** - Essential functionality needed
- **EVE integration** - How it connects to EVE Online systems
- **Similar tools** - Existing alternatives and their limitations

## üè∑Ô∏è Labels and Organization

### Using Labels
We use labels to organize and prioritize issues. You don't need to add labels yourself - maintainers will apply appropriate labels during triage.

### Issue Lifecycle
1. **Triage** - New issues are reviewed and labeled
2. **Planning** - Accepted issues are added to project boards
3. **Development** - Issues move to "in progress" when work begins
4. **Review** - Completed work is reviewed and tested
5. **Closed** - Issues are closed when resolved or declined

## üí¨ Communication Guidelines

### Be Respectful
- Treat all community members with respect
- Use constructive language when providing feedback
- Focus on the issue, not the person
- Welcome newcomers and help them get started

### Be Clear and Concise
- Use clear, descriptive titles and descriptions
- Provide context for your suggestions or reports
- Ask questions if something is unclear
- Update issues with new information as needed

### Stay On Topic
- Keep discussions relevant to the issue at hand
- Use GitHub Discussions for general conversation
- Create new issues for separate problems
- Reference related issues when appropriate

## üîí Security Issues

If you discover a security vulnerability, please do NOT open a public issue. Instead:

1. Email details to [me@mrakaki.com](mailto:me@mrakaki.com)
2. Include steps to reproduce the vulnerability
3. Provide any relevant technical details
4. Wait for confirmation before disclosing publicly

We take security seriously and will respond promptly to legitimate reports.
